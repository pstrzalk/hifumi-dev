## Research Question

Gather knowledge to start Phase 3 — what does the existing analysis say, what is the current state of the generator app, and where will Phase 3 plug in?

## Summary

Phase 3 (preview isolation via Kamal + Docker) is **analysis-ready**, not started. The plan lives in [`docs/03-plans/02-phase-3-preview-isolation.md`](../../../../docs/03-plans/02-phase-3-preview-isolation.md) and proposes: keep Kamal for deploying the generator itself, use `kamal-proxy` directly (without the full Kamal deploy cycle) for routing, and use `docker run` directly with locked-down flags for short-lived per-project preview containers. A `PreviewManager` Ruby class is sketched. SQLite per-container is the natural DB isolation story.

The Rails generator app is sitting at the end of Phase 2: a working chat → CreatePlan → ExecuteInstructionJob → bin/roast → workspace pipeline (3-revision todo-list E2E green on project 38; integration test gated by `E2E_GENERATE=1`). Workspaces live **outside** the generator repo at `~/projects/rails-app-generator-workspaces/project_<id>/` (overridable via `RAILS_APP_GENERATOR_WORKSPACE_ROOT`). The W3 (Preview) workflow is specified in canon as fully deterministic (no LLM steps), and the codebase has no preview-related code yet — no `PreviewManager`, no preview columns on `Project`, no iframe slot in views, no `preview.*` events, no `kamal-proxy` integration. A Rails-generated `Dockerfile` and `config/deploy.yml` exist but were generated by `rails new` and are **not currently active** — Phase 2 used `--skip-kamal` and the generator runs from `bin/dev`.

The Phase 1 spike's `skip_permissions!` on Claude CLI ([`spikes/roast/findings.md:111`](../../../../spikes/roast/findings.md)) is the explicit motivation tying preview isolation to a security requirement: the workspace runs untrusted LLM-generated code with full permissions, so a container boundary is the gate before this can be exposed beyond the developer's own machine.

## Detailed Findings

### Phase 3 design as documented (the analysis)

[`docs/03-plans/02-phase-3-preview-isolation.md`](../../../../docs/03-plans/02-phase-3-preview-isolation.md) contains a full architecture sketch. Key elements:

- **Kamal split** ([lines 9-40](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  - Kamal (full) for deploying the generator app itself.
  - `kamal-proxy` (the standalone Go binary, invoked directly) for hostname routing of preview subdomains.
  - `docker run` direct (NOT through Kamal's deploy pipeline) for spawning preview containers, because Kamal's full build → push → pull → run cycle is too slow per iteration.

- **Architecture diagram** ([lines 44-74](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  ```
  Internet → kamal-proxy (host, 443) → {generator app, preview-1, preview-2, ...}
  All previews on Docker network "preview-internal" (--internal = no egress)
  ```

- **Routing model** ([lines 84-104](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  - DNS wildcard `*.preview.domain.com` → A record to host.
  - kamal-proxy invoked at runtime: `kamal-proxy deploy preview-<id> --target <ip>:3000 --host <id>.preview.domain.com`.
  - Routes added/removed dynamically — no kamal-proxy restart.

- **Docker security flags** ([lines 112-140](../../../../docs/03-plans/02-phase-3-preview-isolation.md)): `--memory=512m --cpus=0.5 --pids-limit=100 --cap-drop=ALL --no-new-privileges --network=preview-internal --read-only --tmpfs /tmp:size=64m --tmpfs /app/tmp:size=64m`.

- **PreviewManager skeleton** ([lines 162-232](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  - `start(project)` → `build_image` → `run_container` → `register_route` → `health_check` → updates `project.preview_url` + `project.preview_container_id`.
  - `stop(project)` → `docker stop` + `docker rm` + `kamal-proxy remove` + clears DB columns.
  - `restart(project)` → stop + start.
  - **Critical rule** ([line 236](../../../../docs/03-plans/02-phase-3-preview-isolation.md)): "we never use a Dockerfile from the generated app." A standard Dockerfile (ours) ships at `lib/preview/Dockerfile`.

- **Build optimization** ([lines 259-284](../../../../docs/03-plans/02-phase-3-preview-isolation.md)): A pre-built base image with common gems (Devise, Pagy, Tailwind, etc.) cuts per-project builds from 1-3 min to 10-30 s.

- **Idle cleanup** ([lines 288-297](../../../../docs/03-plans/02-phase-3-preview-isolation.md)): Solid Queue recurring `CleanupIdlePreviewsJob` stops containers idle > `IDLE_TIMEOUT = 30.minutes`.

- **Rollout phases** ([lines 322-341](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  - **PoC**: `docker run` with security flags only, `localhost:{port}`, manual cleanup, no kamal-proxy.
  - **MVP**: kamal-proxy + `--internal` network + PreviewManager + auto-cleanup + base image + 10-preview cap.
  - **Production**: separate network per container, monitoring, multi-server LB, optional gVisor/Firecracker.

- **Solved vs. accepted risks** ([lines 304-318](../../../../docs/03-plans/02-phase-3-preview-isolation.md)):
  - **Solved**: filesystem, process, network, resource, routing, DB (SQLite per container), privilege escalation.
  - **Accepted**: container-to-container visibility on shared network, Docker escape (kernel shared), DoS via container count, cold-start latency.

- **Plan-to-codebase impact** ([lines 346-352](../../../../docs/03-plans/02-phase-3-preview-isolation.md)) — when activation begins, the plan calls for updating:
  - `docs/01-vision/02-user-journey.md` Step 5 (containers instead of raw `rails server`).
  - `docs/02-architecture/01-workflows-and-decisions.md` W3 (add Docker build step).
  - `docs/02-architecture/03-tech-stack.md` (add Docker / kamal-proxy to generator stack).
  - `CLAUDE.md` (Phase 3 status `analysis` → `active`, add this plan to reading order).

## Architecture Documentation

### Existing patterns the Phase 3 plan inherits from

1. **Subprocess + ENV envelope** — Phase 1 / Phase 2 established that subprocesses to external tooling (Roast / Claude CLI) need a thin wrapper to scrub ENV. The same pattern fits docker invocations: `bin/roast` is the model. The Phase 3 plan uses `system("docker", ...)` directly inside `PreviewManager` ([`docs/03-plans/02-phase-3-preview-isolation.md:181-214`](../../../../docs/03-plans/02-phase-3-preview-isolation.md)); the wrapper-script pattern is available if env hygiene becomes an issue.

2. **Test seam over mocks** — `ExecuteInstructionJob#run_roast_subprocess` ([`app/jobs/execute_instruction_job.rb:126-132`](../../../../app/jobs/execute_instruction_job.rb)) is overridden in tests. PreviewManager will need an analogous seam for `docker run` if jobs are unit-tested without Docker.

3. **Event bus over direct calls** — every cross-layer hop in Phase 2 goes through `ActiveSupport::Notifications` + a subscriber that does only `perform_later` or `broadcast_*`. The Phase 3 design respects this: `instruction.completed` → enqueue `StartPreviewJob` → emit `preview.ready` → broadcast iframe URL.

4. **ENV-overridable filesystem roots** — `RAILS_APP_GENERATOR_WORKSPACE_ROOT` is the established way to relocate per-environment paths. A Phase 3 equivalent for preview-related state (image cache, port range, kamal-proxy socket) would follow the same convention.

5. **Idempotent setup gates** — `ExecuteInstructionJob` uses three explicit gates (`workspace_initialized?`, Gemfile presence, docs/ presence). PreviewManager's `start` is also written idempotent-friendly (`stop` clears DB columns; `restart` is `stop` + `start`).

6. **Solid Queue per-queue concurrency** — `config/queue.yml:6-9` declares the `generation` queue with thread/process count 1. A new `preview` queue would mirror this declaration.

### Patterns the Phase 3 plan introduces (new to the codebase)

- **Per-project Docker container** as a long-lived background process (not a one-shot job). New ops mode for the generator.
- **Standard Dockerfile owned by the generator**, never read from the generated app ([`docs/03-plans/02-phase-3-preview-isolation.md:236`](../../../../docs/03-plans/02-phase-3-preview-isolation.md)) — invariant for security.
- **Two-tier image strategy**: pre-built base + per-project layer ([`docs/03-plans/02-phase-3-preview-isolation.md:259-284`](../../../../docs/03-plans/02-phase-3-preview-isolation.md)).
- **Recurring Solid Queue cleanup job** — first such job in the codebase (`config/recurring.yml:12-15` only has the queue-job janitor today).

## Open Questions

These are the questions explicitly carried in the Phase 3 plan and adjacent docs as still-open at activation time:

1. **Local PoC vs. MVP cut-over** — `docs/03-plans/02-phase-3-preview-isolation.md:324-335` distinguishes "PoC: docker run + localhost:port, no kamal-proxy" from "MVP: kamal-proxy + --internal network + PreviewManager + auto-cleanup". The plan does not commit to which step zero looks like.

2. **Iframe content for `localhost:port` previews** — when the generator runs on a developer machine without a real domain, browsers cannot easily address `<id>.preview.localhost`. Plan PoC mode says `localhost:{port}`; how the iframe `src` is constructed (and whether dev needs `dnsmasq` or `/etc/hosts` tooling) is not specified.

3. **Where preview state lives on `Project`** — the plan's PreviewManager writes `project.preview_url` and `project.preview_container_id` ([`docs/03-plans/02-phase-3-preview-isolation.md:176`](../../../../docs/03-plans/02-phase-3-preview-isolation.md)). The migration shape (also `preview_started_at` for the cleanup job) is not laid out in the plan; the existing `Project` model has none of these columns.

4. **W3 vs. Roast** — current canon (`docs/02-architecture/01-workflows-and-decisions.md:138-150`) describes W3 as a Roast workflow but its steps are deterministic. The Phase 3 plan implements W3 as plain Ruby in `PreviewManager` (no Roast). Whether to keep W3 in canon as "Roast" or document it as a Ruby service is not resolved by the plan.

5. **Selenium / system tests** — memory `project_verify_no_system_tests.md` notes "W2.4 verify can't rely on Selenium/headless Chrome on remote hosts; revisit in Phase 3." This is parked at activation time but not yet scoped.

6. **Deferred-request handling** — `docs/09-ideas/02-deferred-request-handling.md` explicitly names "Phase 3 preview wiring" as a possible activation trigger for picking up A/B/C/D options for what happens to user requests that arrive mid-build.

7. **`kamal-proxy` install path** — the plan invokes `kamal-proxy deploy …` as a binary on the host but does not specify how it is installed alongside the Kamal-deployed generator (Kamal 1.x ships kamal-proxy via its own deploy machinery; standalone install path is its own decision).

8. **Step 7 wall-time slack** — `CLAUDE.md` notes the W2 docs-update agent now consumes most of the 900 s budget; Phase 3 may want to know the budget headroom before adding container-start latency on top.
