# Agents vs Workflows

Boundary: what is an agent (LLM decisions), what is a workflow (defined steps).

## Principle

**Agent** = autonomous decisions. The LLM decides WHAT to do and WHEN.
**Workflow** = a defined sequence of steps. Some steps use the LLM (non-deterministic), but the structure is predetermined.

We minimize "the agent judges for itself." Instead: "workflow W1, step 3 uses the LLM to make decision D with options [a, b, c]."

## Architecture: two layers

```
┌─────────────────────────────────────┐
│  CONVERSATION LAYER (agent)         │
│  RubyLLM + tools                    │
│  Decides: what to tell the user,    │
│  when to trigger a workflow         │
│  → via tool calls                   │
└────────────┬────────────────────────┘
             │ tool call: StartGeneration
             │ tool call: CancelInstruction
             │ tool call: SuggestPrompts
             ▼
┌─────────────────────────────────────┐
│  EXECUTION LAYER (workflows)        │
│  Roast / own orchestration          │
│  Defined steps, traceability,       │
│  session replay                     │
└─────────────────────────────────────┘
```

**Agent** (conversation layer) decides **WHAT** — "the user wants to add a discount system, I trigger the generation workflow."

**Workflow** (execution layer) defines **HOW** — "step 1: load manifest, step 2: research, step 3: plan, steps 4-N: implementation."

Tool calls are the bridge between the agent and the workflows.

## Workflow catalog

### W1: Generating a new application

Triggered by: tool call `StartGeneration(intent, clarifications)`. Revisions are already in the DB when W1 starts — they were generated by the `CreatePlan` service inside the tool handler. W1 receives `instruction_id` and iterates over Revisions.

```
W1.1  [deterministic]  Load archetype (match app type to archetype database)  ← delegated to CreatePlan in Phase 2
W1.2  [LLM]            Domain research (if archetype is insufficient)          ← delegated to CreatePlan in Phase 2
        → Decision D1: does the archetype cover the requirements?
          [a] yes → skip W1.2, proceed to W1.3
          [b] no → LLM analyzes the domain, generates additional context
W1.3  [LLM]            Generate plan (list of revisions)                        ← delegated to CreatePlan in Phase 2
W1.4  [deterministic]  Create Revision records in the database (pending)        ← delegated to CreatePlan in Phase 2
W1.5  [loop]           For each revision → call W2 (Execution workflow)
W1.6  [deterministic]  Mark Instruction as completed
W1.7  [deterministic]  Trigger W3 (Preview)
```

In Phase 2, W1 starts from W1.5 — steps W1.1-W1.4 (archetype, research, plan, revision creation) live inside the `CreatePlan` service invoked in the `StartGeneration` tool handler. In future phases, when the archetype content workstream starts, these steps may return to W1 as a Roast workflow — but they will still remain invisible to the chat LLM.

### W2: Executing a single revision

Triggered by: W1.5 or W4.4 (iteration).

Cycle per revision: **Implement → Verify → Commit**. We don't commit code that doesn't pass verification.

```
W2.1  [deterministic]  Mark revision as generating
W2.2  [deterministic]  Build prompt:
        - step description from the plan
        - app manifest (docs/)
        - revision notes from the previous revision (if any)
        - plan context (what's done, what remains)
W2.3  [LLM/agent]      Execute Claude CLI with the prompt in the workspace cwd
        → Agent (Claude Code) with a constrained scope: step description + cwd.
W2.4  [deterministic]  Verification — sequence of checks:
        a) bundle check              — dependencies resolve
        b) rails db:prepare          — migrations pass
        c) herb lint app/views/      — ERB/HTML correct
        d) rails runner "puts :ok"   — application boots
        e) rails test (if tests exist) — tests pass
        → If all pass → W2.5
        → If fail → W2.R (remediation loop)
W2.5  [deterministic]  Git add + commit (annotated: what and why)
W2.6  [LLM]            Update app manifest (docs/) + revision notes
        → Decision D2: what changed in architecture/conventions/domain?
          LLM generates a diff to docs, doesn't rewrite from scratch.
        → Revision notes: record of implementation decisions for this revision.
          Not a summary ("added models") but context ("Flower uses STI because
          seasonal/permanent have different validation rules, not separate tables").
          Fed into W2.2 of the next revision.
W2.7  [deterministic]  Mark revision as completed, save git_sha
W2.8  [deterministic]  Push status via Turbo Stream
```

#### W2.R: Remediation loop (max 2 attempts)

Verification didn't pass — we hand errors to the agent to fix instead of failing immediately.

```
W2.R1 [deterministic]  Collect output from failed checks (error messages, stack traces)
W2.R2 [LLM/agent]      Claude CLI: fix the problems
        → Prompt: "Verification did not pass. Here are the errors: {errors}. Fix."
        → Same workspace, same cwd. Agent sees its earlier code.
W2.R3 [deterministic]  Re-run verification (W2.4 a-e)
        → pass → continue to W2.5
        → fail and retry_count < 2 → W2.R1
        → fail and retry_count >= 2 → W2.F1 (failure path)
```

Why max 2: the third attempt to fix the same error is "the agent cannot do this", not "minor fix." We escalate to the user.

#### Failure path

```
W2.F1 [deterministic]  Mark revision as failed, save last verification errors
W2.F2 [deterministic]  Git reset --hard to parent revision
W2.F3 [deterministic]  Report failure to the conversation layer
        → Payload contains: what failed (which check), errors, how many attempts
        → Agent (chat) decides how to react (D6): change approach, ask the user
```

#### W2.3 agent prompt invariants

W2.3 hands a prompt to Claude CLI. Three constraints on that prompt must hold; violating any one pushes the agent off-task in ways that burn tokens and are hard to spot until the workspace contains code nobody asked for.

1. **The workspace is a real Rails app.** W2 assumes `rails new`, `git init`, and the `docs/` baseline have already run. An empty directory causes the agent to bootstrap a Rails app itself (it calls `rails new .` because it has nothing to edit). Precondition enforced by the caller — in Phase 2, `ExecuteInstructionJob` runs these once per project before iterating revisions.

2. **No mentions of "Claude" or "Anthropic" in the prompt unless the app genuinely integrates Anthropic.** Claude CLI auto-loads the `claude-api` skill on such mentions, injecting ~8000 lines of Anthropic SDK documentation that then dominate the agent's reasoning. `CreatePlan::AdHocLLM` strips these tokens from revision prompts unless the user's intent explicitly names a Claude/Anthropic integration as the feature to build.

3. **Concrete tasks, not meta.** W2.2 appends a fixed rules block ("Rails Way, Tailwind, Minitest, Write tests for new functionality") to every prompt. The rules assume the Task section names specific code changes ("Add Todo model with title/body/done"). Paired with a vague or meta task ("verify pipeline", "smoke test"), the rules amplify: the agent scaffolds features that weren't requested and writes tests for them. Meta and diagnostic flows do not use `revision_workflow.rb` — they use a dedicated workflow with only `ruby(...)` steps (see `lib/roast/smoke_workflow.rb`), which exercises the wrapper + DSL + require paths without invoking the agent.

These constraints live in the execution layer because they are conditions on what Claude CLI receives. `CreatePlan` (Phase 2) is responsible for producing revision prompts that satisfy them.

### W3: Preview

Triggered by: completion of W1 or W4.

```
W3.1  [deterministic]  Stop existing preview (if any)
W3.2  [deterministic]  bundle install (shared cache)
W3.3  [deterministic]  rails db:prepare
W3.4  [deterministic]  rails server -p {port}
W3.5  [deterministic]  Verify the server responds (health check)
W3.6  [deterministic]  Push preview URL via Turbo Stream
```

W3 is **fully deterministic**. No LLM decisions.

**Implementation**: W3 is implemented as a plain Ruby class (`lib/preview/preview_manager.rb`), not a Roast workflow — every step is deterministic and Roast's value (LLM/agent integration) does not apply. The class is invoked from `StartPreviewJob` / `StopPreviewJob`, both on the `:preview` queue.

**Trigger model in Phase 3**: `start` is user-initiated (button click → `PreviewsController#create`); `stop` is user-initiated OR auto-fired by the `instruction.requested` subscriber (because production-mode containers don't autoload — a running preview shows stale code mid-generation regardless). The `instruction.completed` subscriber does NOT auto-start; preview is a deliberate user-driven feature. A recurring `CleanupIdlePreviewsJob` (every 5 min) stops previews running for >30 min.

### W4: Iteration (change in an existing app)

Triggered by: tool call `StartGeneration(intent, clarifications)` in the context of an existing project. Analogous to W1: revisions are already in the DB (generated by `CreatePlan`), W4 iterates over them.

```
W4.1  [deterministic]  Load app manifest (docs/)                               ← delegated to CreatePlan in Phase 2
W4.2  [LLM]            Research (if needed)                                    ← delegated to CreatePlan in Phase 2
        → Decision D3: are manifest + user request enough for a plan?
          [a] simple (add field, change color) → skip, proceed to W4.3
          [b] requires new solutions → LLM searches for gems, explores options
          [c] requires deeper code understanding than manifest → reads files
W4.3  [LLM]            Generate revision plan                                  ← delegated to CreatePlan in Phase 2
W4.4  [loop]           For each revision → call W2
W4.5  [deterministic]  Mark Instruction as completed
W4.6  [deterministic]  Trigger W3 (restart preview)
```

Same principle as W1: W4.1-W4.3 live inside the `CreatePlan` service. Decisions D1/D3 live in `CreatePlan::AdHocLLM`'s prompt engineering, not in the chat LLM.

### W5: Undo

Triggered by: tool call `UndoLastChange`.

```
W5.1  [deterministic]  Find the last completed revision
W5.2  [deterministic]  Git revert (new commit, not reset)
W5.3  [deterministic]  Create a new Revision with the revert's git_sha
W5.4  [LLM]            Update app manifest (docs/)
W5.5  [deterministic]  Trigger W3 (restart preview)
```

### W6: Cancel

Triggered by: tool call `CancelInstruction` or UI button.

```
W6.1  [deterministic]  Abort the current job (if running)
W6.2  [deterministic]  Mark Instruction as cancelled
W6.3  [deterministic]  Git reset --hard to the last completed revision
W6.4  [deterministic]  Mark pending revisions as cancelled
W6.5  [deterministic]  Report to the conversation layer
```

W6 is **fully deterministic**.

## Decision catalog (non-deterministic steps)

Every place where the LLM makes a decision, explicitly described.

| ID | Workflow | Step | Decision | Options |
|----|----------|------|----------|---------|
| D1 | W1 | W1.2 | Does the archetype cover the requirements? | [a] yes → skip research [b] no → additional research |
| D2 | W2 | W2.6 | What changed in the documentation? | LLM generates a diff for app manifest + revision notes |
| D3 | W4 | W4.2 | How much research is needed? | [a] manifest is enough [b] look for new solutions [c] read code |
| D4 | Chat | — | When to trigger a workflow? | Agent interprets user intent → tool call |
| D5 | Chat | — | What to suggest to the user? | Agent generates suggested prompts |
| D6 | Chat | — | How to react to failure? | Agent decides: change approach / ask the user (has verification errors in context) |

## Agent — the only unstructured element

The conversational agent (RubyLLM + tools) is the only place with truly autonomous decisions:
- Interprets what the user means
- Decides when to ask, when to generate, when to suggest
- Reacts to events (failure, completion) from workflows

But even the agent is constrained:
- Has a defined set of tools — cannot do anything we didn't anticipate
- A system prompt defines its behavior and boundaries
- Does not execute code — triggers workflows

## Roast — the workflow engine

Roast (Shopify) — a Ruby DSL for orchestrating workflows with mixed steps (deterministic + LLM). Session replay, Claude Code integration, production-tested at Shopify.

W2 example in Roast — **kept in sync with the working `../../spikes/roast/revision_workflow.rb`** (validated end-to-end with the `todo-list` and `force-remediation` plans, 2026-04-15/16).

```ruby
# workflows/revision_workflow.rb
# Run: roast revision_workflow.rb -- revision_id=123 revision_summary="..." revision_prompt="..."
# Workspace via ENV: HIFUMI_DEV_WORKSPACE=/path/to/project

WORKSPACE = ENV.fetch("HIFUMI_DEV_WORKSPACE")
CLAUDE_MODEL = ENV.fetch("HIFUMI_DEV_MODEL", "sonnet")

# Shared state between steps. Roast DSL blocks do NOT share `metadata` or
# instance variables — the only reliable way to pass data between steps
# (besides `kwarg` and bang-suffix) is a module-level container.
WORKFLOW_STATE = {}

config do
  agent do
    provider :claude
    model CLAUDE_MODEL
    working_directory WORKSPACE
    skip_permissions!             # batch mode, no interaction
    show_stats!
  end
  cmd { display! }
end

# W2.R — Remediation scope (max 2 attempts)
execute(:fix_and_reverify) do
  agent(:fix) do |_, errors, idx|
    "Verification didn't pass (attempt #{idx + 1}/2). Errors:\n\n#{errors}\n\nFix exactly what's broken."
  end
  ruby(:reverify) do
    result = VerifyRevision.run(WORKSPACE)
    fail!(VerifyRevision.format_errors(result)) if VerifyRevision.failed?(result)
    VerifyRevision.summary(result)
  end
  ruby { |_, _, idx| break! if ruby?(:reverify) || idx >= 1 }
  outputs { ruby?(:reverify) ? :succeeded : :failed }
end

# Main workflow
execute do
  ruby(:log_start) { ... }                        # W2.1 deterministic

  ruby(:build_prompt) { ... }                     # W2.2 gather manifest + notes
  agent(:generate_code) { ruby!(:build_prompt).value }  # W2.3 Claude CLI

  ruby(:verify) do                                # W2.4 check sequence
    result = VerifyRevision.run(WORKSPACE)
    if VerifyRevision.failed?(result)
      errors = VerifyRevision.format_errors(result)
      WORKFLOW_STATE[:verify_errors] = errors     # <- key: preserve errors
      fail!(errors)
    end
    "all checks passed"
  end

  # W2.R remediation — skip if verify is green
  repeat(:remediate, run: :fix_and_reverify) do
    skip! if ruby?(:verify)
    WORKFLOW_STATE[:verify_errors] || "initial verification failed"
  end

  # W2.F failure guard — don't commit if remediation also failed
  ruby(:ensure_passing) do
    passed = ruby?(:verify) || (ruby?(:remediate) && repeat!(:remediate).value == :succeeded)
    unless passed
      system("cd #{WORKSPACE} && git reset --hard HEAD && git clean -fd")
      fail!("W2.F: revision failed, workspace reset to parent HEAD")
    end
    "ready to commit"
  end

  cmd(:git_commit) { ... }                        # W2.5 commit code
  agent(:update_docs) { ... }                     # W2.6 update manifest + revision notes
  cmd(:git_commit_docs) { ... }                   # W2.7 commit docs (allow-empty)
  ruby(:report) { ... }                           # W2.8 report + git_sha
end
```

**Gotchas surfaced in the spike (to keep):**

- `metadata[:key] = ...` inside `ruby(:...)` blocks **blows up** with `NameError: undefined local variable 'metadata' for Roast::CogInputContext`. Workaround: module-level `WORKFLOW_STATE = {}`.
- `ruby(:verify).error` after `fail!` **is not available** — errors must be captured manually before `fail!`.
- An agent without `skip_permissions!` times out in batch mode (prompts for `--dangerously-skip-permissions`). Acceptable because the workspace is isolated — but confirms the need for preview isolation (containers) in production.
- W1 does not call W2 natively via Roast composition. Pattern: a **Ruby wrapper** (`new_app_driver.rb` in the spike — equivalent of the future Solid Queue job) invokes `bin/roast revision_workflow.rb` per revision via `system()`. ENV hygiene (`HIFUMI_DEV_WORKSPACE`, `HIFUMI_DEV_MODEL`) passed via env, per-revision parameters via `kwarg=value`.

## Technology split

Three layers, clear boundaries:

```
┌─────────────────────────────────────────┐
│  RubyLLM                                │
│  Communication with user and LLM        │
│  Chat, tools, suggested prompts         │
│  → tool call triggers Solid Queue job   │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│  Solid Queue                            │
│  Junction of the layers                 │
│  Job runs workflow, broadcasts          │
│  Turbo Stream on completion             │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│  Roast                                  │
│  Server-side orchestration              │
│  Building the client application (W1-W6)│
│  Claude CLI, git, files, docs           │
└─────────────────────────────────────────┘
```

**RubyLLM** doesn't know how to build applications. It knows how to talk to the user and when to trigger workflows.

**Roast** doesn't know what the user said. It receives an Instruction from the DB and executes defined steps.

**Solid Queue** links the two layers asynchronously.

## Summary

| Layer | Type | Tool | Decisions |
|-------|------|------|-----------|
| Conversation | Agent | RubyLLM + tools | D4, D5, D6 — user interpretation, suggestions, reactions |
| New app generation | Workflow W1 | Roast | D1 — whether extra research is needed |
| Revision execution | Workflow W2 | Roast | D2 — update docs + revision notes. Verify + remediation loop (max 2 attempts) |
| Preview | Workflow W3 | Roast | none — fully deterministic |
| Iteration | Workflow W4 | Roast | D3 — how much research |
| Undo | Workflow W5 | Roast | none beyond update docs |
| Cancel | Workflow W6 | Roast | none — fully deterministic |
