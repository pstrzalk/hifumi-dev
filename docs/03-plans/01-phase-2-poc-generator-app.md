# Phase 2 вҖ” PoC of the main generator app

Rails skeleton + RubyLLM chat + a Solid Queue job running the proven workflow from Phase 1 (`../../spikes/roast/revision_workflow.rb`).

## Goal

Prove end-to-end that the pipeline from the spike runs from a Rails app instead of `new_app_driver.rb`. The driver becomes `ExecuteInstructionJob`, hardcoded `plans.rb` вҖ” Instruction + Revision in the DB created by a tool call from the RubyLLM chat (via the `CreatePlan` service).

Phase 1 showed that Roast + Claude CLI work. Phase 2 shows that we can wrap it in an app aligned with `../01-vision/02-user-journey.md` and `../02-architecture/01-workflows-and-decisions.md`.

## Architectural decisions (confirmed 2026-04-16)

We went through alternatives A1-A7. Most confirmed the plan's default; two architectural changes + one scope refinement.

| # | Decision | Change vs. default |
|---|----------|--------------------|
| A1 | **Subprocess `bin/roast`** вҖ” wrapper neutralizes 3 ENV gotchas | = plan |
| A2 | **Claude CLI** as agent in Roast; `bin/roast-openrouter` fallback | = plan |
| A3 | **Roast** as orchestrator; `revision_workflow.rb` moved 1:1 | = plan |
| A4 | **Local FS** (`storage/workspaces/<id>/`) in PoC; **in production user isolation вҶ’ Phase 3** (explicit requirement, not just a preview consequence) | = plan + explicit Phase 3 TODO |
| A5 | **One `ExecuteInstructionJob`** with a loop over revisions; chaining per Revision as a future extension | = plan |
| A6 | **`CreatePlan` service as abstraction** with the first implementation `CreatePlan::AdHocLLM`; swappable (archetype, hybrid, cheap-but-good model) in the future. Plan quality = key to generator quality, but a separate workstream outside Phase 2 | **CHANGE**: abstraction layer added, tool does not create the plan directly |
| A7 | **Lightweight tool `StartGeneration(intent, clarifications)`** вҖ” detailed prompts do NOT leave through the chat API. Secret sauce (planner prompt engineering) lives inside `CreatePlan`, not in the chat system prompt nor in tool call args | **CHANGE**: tool passes intent instead of a completed plan |

**Consequences for the rest of the plan**:
- Step 4: the tool is called `StartGeneration`, delegates to `CreatePlan.call(intent, clarifications)`, which returns Revisions
- The chat system prompt does not contain rules like "Rails Way, 3-6 steps, Tailwind" вҖ” those rules live inside `CreatePlan::AdHocLLM`'s internal prompt
- Instruction model: `description` generated by `CreatePlan` (human-readable, git-commit-able); consider a separate `user_intent: text` for raw input from chat (to be decided in Step 4)

## Definition of Done

The app satisfies all of the below:

1. Rails 8 + RubyLLM + Solid Queue + Solid Cable + Tailwind stands up on `bin/rails s`
2. User types an app description вҶ’ chat responds (RubyLLM chat.ask without tools, baseline)
3. After the second-third exchange the LLM calls `StartGeneration(intent, clarifications)` tool вҶ’ `CreatePlan` service generates revisions вҶ’ `Instruction` + `Revision` Г— N appear in the DB
4. Solid Queue `ExecuteInstructionJob` runs `bin/roast` with the moved `revision_workflow.rb` in the project workspace cwd
5. In `storage/workspaces/<project_id>/` a real Rails app appears with git history вҖ” one revision = one commit
6. Revision status (generating/completed/failed) flows through Turbo Stream to the chat UI
7. After `instruction.completed`, a subscriber enqueues `ChatFollowUpJob` вҶ’ LLM invokes `SuggestPrompts` вҶ’ user sees cards with proposals (prompts may be hints for chat, but the detailed plan for the next revision is still generated by `CreatePlan`)
8. Demo passes on a plan triggered from the chat, analogous to `TODO_LIST` from the spike (3 revisions Г— Sonnet, green `rails test`, ~8 minutes wall)
9. CLI mirror: `bin/generate full --prompt "..."` does the same without UI вҖ” for debugging and integration tests

**Watch ENV hygiene** вҖ” three gotchas from `feedback_roast_rails_env_gotchas` are still current:

1. `ANTHROPIC_API_KEY` leak вҶ’ `bin/roast` wrapper still unsets it. The Rails app has no reason to export the key, but someone might have it in `.envrc` вҖ” the wrapper neutralizes it.

In `ExecuteRevisionJob` additionally: the Roast subprocess gets clean ENV (without `BUNDLE_*`), then Roast initializes under its own Gemfile (the generator's), and verify inside the workflow shells to the workspace with clean env.

- Tool `app/tools/start_generation.rb`:
  ```ruby
  class StartGeneration < RubyLLM::Tool
    description "Starts application generation. Call when the user has described what they want and you are ready."
    param :intent, type: :string, desc: "Plain language: what the user wants to build, e.g. 'flower shop with inventory and Stripe'"
    param :clarifications, type: :object, desc: "Answers to clarifying questions: { key: value }"

    def execute(intent:, clarifications: {})
      project = Current.project
      revisions_data = CreatePlan.call(
        intent: intent,
        clarifications: clarifications,
        context: { project_id: project.id }
      )

      instruction = project.instructions.create!(
        user_intent: intent,            # raw from chat, for audit
        description: revisions_data.first[:instruction_description] || intent.truncate(200),
        phase: :processing,
        anchor_message: project.chat.messages.last
      )
      revisions_data.each_with_index do |r, i|
        instruction.revisions.create!(
          project: project,
          summary: r[:summary],
          prompt: r[:prompt],           # detailed prompt вҖ” lives only in DB and ExecuteInstructionJob
          position: i + 1,
          status: :pending,
          parent: i == 0 ? nil : instruction.revisions[i - 1]
        )
      end
      ActiveSupport::Notifications.instrument("instruction.requested", instruction_id: instruction.id)

      # IMPORTANT: only a high-level confirmation goes back to chat, not prompts
      { instruction_id: instruction.id, revision_count: revisions_data.size, intent: intent }
    end
  end
  ```

### Step 5 вҖ” `ExecuteInstructionJob` + Roast integration (1-2 days)

This is the meatiest part. Moves all the logic from `new_app_driver.rb` into a Rails job with status persistence.

```ruby
class ExecuteInstructionJob < ApplicationJob
  def perform(instruction_id)
    instruction = Instruction.find(instruction_id)
    project = instruction.project
    workspace = Rails.root.join("storage/workspaces/#{project.id}").to_s

    prepare_workspace(project, workspace) unless project.workspace_initialized?
    rails_new(project, workspace) unless File.exist?(File.join(workspace, "Gemfile"))
    init_docs_baseline(workspace) unless File.exist?(File.join(workspace, "docs"))

    instruction.revisions.order(:position).each do |revision|
      execute_revision(revision, workspace)
      break if revision.failed?
    end

    instruction.update!(phase: instruction.revisions.all?(&:completed?) ? :completed : :failed)
    ActiveSupport::Notifications.instrument(
      instruction.completed? ? "instruction.completed" : "instruction.failed",
      instruction_id: instruction.id
    )
  end

  private

  def execute_revision(revision, workspace)
    revision.update!(status: :generating, started_at: Time.current)
    ActiveSupport::Notifications.instrument("revision.started", revision_id: revision.id)

    env = {
      "REVISION_WORKSPACE" => workspace,
      "CLAUDE_MODEL" => ENV.fetch("CLAUDE_MODEL", "sonnet")
    }
    args = [
      Rails.root.join("bin/roast").to_s,
      Rails.root.join("lib/roast/revision_workflow.rb").to_s,
      "--",
      "revision_id=#{revision.id}",
      "revision_summary=#{revision.summary}",
      "revision_prompt=#{revision.prompt}"
    ]

    started = Time.current
    ok = system(env, *args)
    exit_code = $?.exitstatus
    wall = Time.current - started

    metrics = {
      wall_seconds: wall.round(2),
      exit_code: exit_code,
      git_sha: git_head(workspace)
    }

    if ok
      revision.update!(status: :completed, finished_at: Time.current, git_sha: metrics[:git_sha], metrics: metrics)
      ActiveSupport::Notifications.instrument("revision.completed", revision_id: revision.id, git_sha: metrics[:git_sha])
    else
      revision.update!(status: :failed, finished_at: Time.current, metrics: metrics)
      ActiveSupport::Notifications.instrument("revision.failed", revision_id: revision.id, error: "exit #{exit_code}")
    end
  end

  # rails_new, init_docs_baseline, git_head вҖ” cut/paste from new_app_driver.rb
end
```

- `app/jobs/application_job.rb`: `queue_as :generation` + `retry_on StandardError, wait: :polynomially_longer, attempts: 1` (we don't auto-retry generation вҖ” after a fail the user decides)
- Solid Queue config: `generation` queue with concurrency 1 (one generation at a time, per ../01-vision/02-user-journey.md В§ "Parallel instructions" вҖ” one active)

## Risks and what to do about them

| Risk | Mitigation |
|------|------------|
| RubyLLM calls StartGeneration inconsistently | System prompt with explicit `You must call StartGeneration` + few-shot. Fallback: UI button "Start generating" that forces a tool call with chat content as intent |
| `CreatePlan::AdHocLLM` generates plans of uneven quality | We log every generation (intent вҶ’ revisions) to DB. After 5-10 runs analyze вҖ” if quality varies, decide whether to implement `CreatePlan::Archetypes` or a hybrid. Fallback is not a blocker for the PoC |
| `bin/roast` subprocess hangs | Step 7 watches; we add a hard 20min timeout if it happens more than 1/10. Earlier: Solid Queue has a configurable job timeout, more diagnostics than blocker |
| RubyLLM Chat/Message schema conflicts with our own columns | First run `rails generate ruby_llm:install`, then write Instruction/Revision migrations. If we need a custom field on Message вҖ” extension model instead of a column |
| Solid Queue + long subprocess blocks a worker | Concurrency=1 on the `generation` queue is OK (one instruction at a time by definition). If it blocks other queues too вҶ’ a separate worker process for `generation` |
| Claude CLI rate limit on subscription | Spike showed 3 revisions Г— Sonnet are OK. With a demo for two users at once the limit may be hit вҶ’ reject the second instruction with retry-after, not parallelism |
| Absolute workspace path in DB вҶ’ migratability | In the DB we keep **relative** (e.g. `storage/workspaces/123`), absolutize in the job via `Rails.root.join`. On restore/host change reset workspace_path, workspace recreate is acceptable (everything is in git) |
| `rails new` in subprocess overwrites generator's `.ruby-version` | `rails_new` is called with `chdir: workspace_root`, and the command itself creates a subdirectory. The generator has `.ruby-version` in the root, the workspace gets its own. But **NOTE**: check that `bundle` in the workspace doesn't use the generator's frum shim вҖ” `VerifyRevision.with_clean_bundler_env` should handle it (verified in the spike) |

## Open questions (to decide along the way)

1. **`CreatePlan::AdHocLLM` implementation**: does the LLM call in the service use RubyLLM with `response_format: json_schema` (deterministic output) or a second tool use? Decided in Step 4. Schema vs. tool is a tradeoff between deterministic parsing and ease of multi-step reasoning
2. **Workspace per project vs. shared?** Per project (`storage/workspaces/<id>`). Shared bundle cache (`~/.bundle`) so we don't reinstall gems on every `rails new` вҖ” to be optimized in Step 5 if Step 7 wall time exceeds the limit
3. **Process supervisor for the Roast subprocess?** Start: plain `system()`. If it turns out we need timeout + kill + PID tracking вҖ” minimum `Process.spawn` + watchdog thread. Full supervisor (e.g. Dragonfly) only when Phase 2.5 (cancel) starts
4. **Does the RubyLLM chat receive context about `Project.revisions`?** Happy-path says yes. In Phase 2 PoC: yes, via `chat.with_instructions(...)` set in `ChatRespondJob` each time (not in `Chat.create!`) to refresh state. Format: short markdown summary with a list of revision summaries + statuses (summaries OK вҖ” they're user-facing; detailed prompts are still hidden from chat)
5. **Which model for `CreatePlan::AdHocLLM`?** Sonnet/Haiku? Sonnet gives better plans, Haiku cheap/fast. Experiment in Step 4. Swappable architecture (A6) allows easy testing of both

## Alternatives (for possible return)

Every decision from the "Architectural decisions" section above is reversible. The table signals when it's worth returning to the alternative. Full reasoning behind these choices вҖ” in git history (commit of the plan rollout, 2026-04-16).

| # | Chosen | Alternative | Return when |
|---|--------|-------------|-------------|
| A1 | Subprocess `bin/roast` | Native Ruby embedding (`require "roast"` in-process) | Subprocess startup becomes a significant % of revision time, or we want live Turbo Stream from the workflow without subprocess pipe |
| A2 | Claude CLI (subscription) | RubyLLM + our own tools (WriteFile/RunBash/ReadFile) in a single process | Subscription no longer covers cost, or we want to let users bring their own Anthropic/OpenRouter keys |
| A3 | Roast as orchestrator | Plain Ruby job (`build_prompt вҶ’ cli вҶ’ verify вҶ’ remediation вҶ’ commit`) | Roast DSL hinders more than it helps (3 gotchas from the spike are already a signal), or requirements appear beyond its reach (real-time streaming, branching) |
| A4 | Local FS `storage/workspaces/<id>/` | `Dir.mktmpdir` per instruction, or a Docker container per project | We move beyond dev-only (multi-user вҶ’ isolation becomes necessary), or we test the generator in CI (tmpdir) |
| A5 | One `ExecuteInstructionJob` with revision loop | `ExecuteRevisionJob` chained per revision (better observability, granular retry) | We want to retry a single failed revision without rerunning the whole Instruction |
| A6 | `CreatePlan::AdHocLLM` behind the `CreatePlan` abstraction | `CreatePlan::Archetypes` (templates + slot-filling), or hybrid | Ad-hoc plan quality turns out to vary (logs from 5-10 runs), the archetype content workstream kicks off, or we find a cheap-but-good model (Haiku? GPT-4.1 mini?) |
| A7 | Lightweight tool `StartGeneration(intent, clarifications)` | `CreateInstruction(description, revisions: [...])` with a detailed plan in tool args, or `response_format: json_schema` | Tool reliability becomes a problem вҖ” LLM doesn't call `StartGeneration` when expected. Fallback: UI button "Start generating" forces the tool call |
